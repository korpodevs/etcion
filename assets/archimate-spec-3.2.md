# Archimate Specification Version 3.2

## 1. Introduction

- **File Path**: `archimate-spec-3.2/ch-introduction.html`
- **Semantic Summary**: ArchiMate is a visual Enterprise Architecture modeling language that provides a uniform set of entities, relationships, and iconography for describing, analyzing, and communicating architecture concerns as they change over time. It organizes architecture across layered domains (Business, Application, Technology) using service-orientation and realization relationships, and separates model elements from their notation to support varied, stakeholder-oriented views.
- **Key Entities**: Architecture Description, Enterprise Architecture, stakeholder, view, viewpoint, layer, aspect, service orientation, realization relationship, language customization mechanism
- **Logic & Constraints**: A conforming implementation shall support: language structure and generic metamodel (Ch. 3-4); all element types across layers and cross-layer dependencies (Ch. 5-12); standard iconography (Ch. 4-12, Appendix A); viewpoint mechanism (Ch. 13); language customization mechanisms (Ch. 14); and the normative relationships table (Appendix B). Support for example viewpoints (Appendix C) is optional (may). Terminology keywords carry precise normative weight: "shall" = requirement, "shall not" = absolute prohibition, "should" = recommended, "may" = optional, "can" = possible, "will" = same as shall (but "shall" is preferred).
- **Requirements Context**: The conformance list in section 1.3 is the minimum feature surface the Python library must cover to be spec-compliant. Terminology definitions in section 1.5 govern interpretation of every normative statement in the spec and should inform how the library documents and enforces its own validation rules. The separation of model elements from notation is an architectural signal that the library should keep its data model and rendering concerns decoupled.

## 2. Definitions

- **File Path**: `archimate-spec-3.2/ch-definitions.html`
- **Semantic Summary**: This chapter provides the authoritative formal definitions for all foundational ArchiMate concepts, establishing a precise vocabulary that governs the interpretation of every normative statement in the remainder of the specification. Undefined terms are delegated by normative reference to the TOGAF framework first, then to Merriam-Webster's Collegiate Dictionary (11th Edition), with any conflict between ArchiMate definitions and TOGAF declared unintentional.
- **Key Entities**: ArchiMate Core Framework, ArchiMate Core Language, Architecture View, Architecture Viewpoint, Aspect, Attribute, Composite Element, Concept, Conformance, Conforming Implementation, Core Element, Element, Layer, Model, Relationship, Relationship Connector
- **Logic & Constraints**: Concept is a union type: exactly one of Element, Relationship, or Relationship Connector. Composite Element is distinguished from a plain Element by its cross-aspect or cross-layer composition. Relationship is further classified into four subtypes: structural, dependency, dynamic, or other. A Relationship Connector connects two or more relationships and only relationships of the same type. Conforming Implementation is defined by reference to Section 1.3 (the conformance clause), making that clause the normative gate for library compliance. The ArchiMate Core Language is scoped to three layers only (Business, Application, Technology including Physical); the ArchiMate Core Framework classifies elements within those layers using three aspects.
- **Requirements Context**: The union definition of Concept (Element | Relationship | RelationshipConnector) is the natural root type for the Python class hierarchy. Element and Relationship must be modeled as distinct base classes; Relationship Connector is a first-class type, not a subtype of either. Attribute is defined as a property on both elements and relationships, indicating that a shared attribute mixin or base class is appropriate. Model is formally a collection of concepts, which informs the top-level container class. The four Relationship subtypes (structural, dependency, dynamic, other) suggest an enum or class hierarchy for relationship classification. Conformance is defined as fulfillment of specified requirements, pointing the library toward a dedicated validation layer that can be tested against the Section 1.3 checklist.

## 3. Language Structure

- **File Path**: `archimate-spec-3.2/ch-language-structure.html`
- **Semantic Summary**: This chapter defines the structural skeleton of the ArchiMate language: a top-level concept hierarchy (model → concept → element | relationship), a three-layer core (Business, Application, Technology) organized across three aspects (Active Structure, Behavior, Passive Structure) forming the nine-cell ArchiMate Core Framework, and an extended Full Framework that adds Strategy, Motivation, Physical, and Implementation & Migration concerns. It also establishes normative rules for abstraction levels, notation conventions, nesting semantics, and the separation of language concepts from their graphical representation.
- **Key Entities**: ArchiMate Core Framework, ArchiMate Full Framework, Business Layer, Application Layer, Technology Layer, Strategy Layer, Motivation Aspect, Implementation & Migration Layer, Physical elements, Active Structure Aspect, Behavior Aspect, Passive Structure Aspect, behavior element, structure element, motivation element, composite element, serving relationship, realization relationship, grouping, abstraction levels (conceptual/logical/physical), nesting, layer letter notation (M S B A T P I)
- **Logic & Constraints**: The top-level type hierarchy is strict and abstract: a concept is exactly one of element or relationship; element is exactly one of behavior element, structure element, motivation element, or composite element — none of these abstract types are directly instantiable in models. The Core Framework is a 3×3 classification grid (three layers × three aspects); element classification within it is global and approximate — elements are not strictly confined to one cell. A composite element by definition spans more than one aspect. The Full Framework extends the Core with Strategy, Motivation, Physical, and Implementation & Migration but does not add a dedicated information layer; information is represented via passive structure elements (business objects, data objects, artifacts) across layers. The language intentionally does not distinguish types from instances at the EA abstraction level. Nesting is a valid alternative graphical notation for certain relationships (defined fully in Chapter 5). Color has no formal semantics in models; in spec diagrams, grey shading encodes aspect (white = abstract, light grey = passive structure, medium grey = behavior, dark grey = active structure); the standard layer-color convention is yellow = Business, blue = Application, green = Technology. Corner shape encodes element category in standard notation: square = structure, round = behavior, diagonal = motivation. Layer-letter badges (M, S, B, A, T, P, I) are a valid alternative to color for layer identification. The serving relationship is the primary cross-layer dependency; realization relationships link elements across abstraction levels (e.g., artifact realizes data object, data object realizes business object).
- **Requirements Context**: The abstract element type hierarchy maps directly to the Python class hierarchy: `Concept` as root, `Element` and `Relationship` as first-level abstract subclasses, then `BehaviorElement`, `StructureElement`, `MotivationElement`, and `CompositeElement` as abstract intermediate classes — none of these should be directly instantiable. The Core Framework grid (layer × aspect) should be encoded as metadata on concrete element classes, not as a rigid containment hierarchy, since classification is approximate. The Full Framework's additional layers and aspects must be first-class citizens alongside the core three layers. The type/instance non-distinction is a deliberate scope boundary: the library models types, not runtime instances. Nesting as a notation alias for relationships means the data model must support a rendering hint on relationships distinct from the relationship's semantic type. Color and corner-shape conventions are presentation metadata and belong in a rendering/notation layer, not in the core data model.

## 4. Generic Metamodel

- **File Path**: `archimate-spec-3.2/ch-generic-metamodel.html`
- **Semantic Summary**: This chapter defines the layer-independent generic metamodel for ArchiMate elements, establishing the abstract type hierarchy of structure elements (active and passive) and behavior elements (internal, external/service, and event) that all layer-specific elements specialize, along with the generic rules governing how active structure, behavior, and passive structure elements relate to one another. It also introduces motivation elements (the "why" behind architecture) and composite elements (Grouping and Location) as cross-cutting concepts that span multiple aspects or layers.
- **Key Entities**: Internal Active Structure Element, External Active Structure Element (Interface), Collaboration, Internal Behavior Element, External Behavior Element (Service), Event, Process, Function, Interaction, Passive Structure Element, Motivation Element, Composite Element, Grouping, Location
- **Logic & Constraints**: All white-box elements in the metamodel hierarchy are abstract and cannot be directly instantiated in models; only their layer-specific descendants are instantiable. Structure elements subdivide into active (subjects that perform behavior) and passive (objects on which behavior is performed); passive structure elements cannot themselves perform behavior. Active structure elements subdivide into internal (the performers) and external/interface (access points that expose services to the environment). Behavior elements subdivide into internal behavior (performed by one or more active structure elements), external behavior/service (explicitly exposed behavior), and event (a state change, which may carry a time attribute). An interaction is a specialized internal behavior element that requires two or more active structure elements (directly assigned or via a collaboration); a collaboration is the corresponding active structure aggregate of two or more internal active structure elements. Processes and functions are both specializations of internal behavior element: a process is sequence-oriented (achieves a specific result), a function is criteria-based (grouped by resources, competencies, or location). Within a layer, composition and aggregation between processes, functions, and interactions are all permitted — a process may be composed of functions and vice versa. Every element in the language may have composition, aggregation, and specialization relationships with elements of the same type (this is implicit and not fully shown in Figure 5). The normative and complete set of permitted relationships is deferred to Appendix B. Metamodel figures show metamodel-level relationships (inheritance, composition within the metamodel), not ArchiMate modeling relationships. Motivation elements are distinct from structure/behavior elements and use diagonal-corner notation; their full taxonomy (stakeholder, driver, assessment, goal, outcome, principle, requirement, constraint, value, meaning) is detailed in Chapter 6. Composite elements (Grouping, Location, and layer-specific Product and Plateau) may span aspects and layers; a grouping links to its members via aggregation or composition relationships, and a concept may belong to multiple overlapping groups. Grouping is not equivalent to an architecture view; it provides in-model structural organization, not a separate visualization. A location aggregates structure elements (active or passive) to indicate where they reside and may also aggregate behavior elements to indicate where behavior is performed.
- **Requirements Context**: The abstract/concrete distinction in the metamodel is a direct constraint on the Python class hierarchy: Internal Active Structure Element, Internal Behavior Element, Passive Structure Element, and Motivation Element must be abstract base classes; concrete instantiable classes exist only at the layer-specific level. The active-structure → behavior → passive-structure directionality (active elements perform behavior; behavior elements access passive elements) is the primary assignment/access rule and should be enforced at validation time. The Process/Function/Interaction distinction within internal behavior elements suggests a further sub-hierarchy or discriminated type that the library needs to represent. An Interaction's multiplicity constraint (requires ≥ 2 active structure elements) and Collaboration's corresponding aggregate structure are runtime validation rules. The time attribute on Event is the first example of an element-specific optional attribute and indicates the library needs per-type attribute schemas. Grouping's ability to contain both elements and relationships of mixed types means the library's container model must accept `Concept` (not just `Element`) as members.

## 5. Relationships and Relationship Connectors

- **File Path**: `archimate-spec-3.2/ch-relationships-and-relationship-connectors.html`
- **Semantic Summary**: This chapter defines the complete set of generic ArchiMate relationships, classifying them into four categories — structural (static composition), dependency (support/usage), dynamic (temporal/causal), and other (specialization) — along with the Junction relationship connector that joins relationships of a single type. It establishes that all permitted relationships between any two element types are derivable from chains of modeled relationships, enabling models to elide intermediary elements while still making valid architectural statements.
- **Key Entities**: Composition, Aggregation, Assignment, Realization, Serving, Access, Influence, Association, Triggering, Flow, Specialization, Junction (and-junction, or-junction)
- **Logic & Constraints**: Each relationship has exactly one "from" and one "to" concept (element, relationship, or relationship connector). No relationships are allowed between two relationships. All relationships joined by relationship connectors must be of the same type. A chain of same-type relationships connected via junctions is valid only if a direct relationship of that type between the two endpoint elements is also valid. A relationship connecting an element with a second relationship may only be an aggregation, composition, or association; aggregation or composition in this case is valid only from a composite element. Composition and aggregation are always permitted between two elements of the same type. Specialization is always permitted between two elements of the same type. Association is always permitted between any two elements and between any element and relationship. The complete normative set of permitted relationships is in Appendix B. Structural relationships (Composition, Aggregation, Assignment, Realization) represent static coherence; they are transitively applicable to unmodeled parts of the source element: aggregation/composition from a part also applies to the whole; assignment to a behavior element also applies to the active structure element; realization of a service also applies to the realizing process. Assignment direction is always active structure → behavior or behavior → passive structure or active structure → passive structure; the non-directional (double-ball) notation from ArchiMate 2.1 is deprecated. Structural relationships may alternatively be expressed by nesting the target concept inside the source element. Access relationship direction at the metamodel level is always from a behavior or active structure element to a passive structure element; the arrowhead in notation may point toward the accessor (read) or in both directions (read-write), but this notation direction is independent of metamodel directionality and must not be used in derivations. Influence carries optional sign/strength attributes (e.g., ++, +, 0, -, -- or a numeric range); the relationship models positive contribution or negative impact; realization should be used instead when the relationship is critical to the target's existence. Association is undirected by default but may be directed; directed association can only be used in derivations in the stated direction. Triggering semantics follow structural semantics: triggering A→B means everything in B is preceded by a part of A. Flow semantics follow dependency semantics: flow A→B means some part of A transfers something to some part of B; flow does not imply causality. Specialization semantics: the whole of the generic element is specialized by the specializing element. Junction rules: a junction connects relationships of the same type; it may have multiple incoming and one outgoing, one incoming and multiple outgoing, or multiple incoming and multiple outgoing (the last form is shorthand for two contiguous junctions); an and-junction requires all participants; an or-junction requires at least one participant (inclusive or exclusive, indicated informally by naming); a junction aggregated or composed into a plateau, grouping, or location should be interpreted without that containing relationship; arrowheads on relationships leading into a junction may be omitted; labels on outgoing triggering relationships of a junction are informal guards only (no formal operational semantics). Derivation rules: indirect relationships between elements may be derived from chains of modeled relationships (e.g., assignment → composition → realization → serving yields a derived serving); derivation is valid only in the direction from more detail to less detail (summarization/abstraction); derived relationships may be modeled directly without the constituent chain being present; from a derived relationship alone no conclusions can be drawn about the specific chain that produced it. The precise derivation rules are normative and specified in Appendix B.
- **Requirements Context**: The library must represent all eleven relationship types plus Junction as first-class types with a shared abstract Relationship base. Each relationship class must carry its source and target typed as `Concept` (to allow element-to-element, element-to-relationship, and connector involvement), but should enforce type-specific endpoint constraints at validation time per Appendix B. The four relationship categories (structural, dependency, dynamic, other) should be an enum or mixin on the Relationship base class. Access relationship must store an access-mode attribute (read, write, read-write, unspecified) because the notation direction is decoupled from metamodel direction; derivation logic must use metamodel direction, not notation direction. Influence relationship must store optional sign and strength attributes. Association must store a direction attribute (undirected, directed-from, directed-to). Junction must be modeled as a RelationshipConnector, not a Relationship; it requires a type attribute (and/or) and enforces that all connected relationships share the same relationship type. The derivation engine must implement chain-traversal rules from Appendix B; it must treat derivation as a summarization operation (many-to-one, lossy), not an inference engine — a derived relationship cannot be unpacked to recover its chain. Nesting notation should be stored as a rendering hint on structural relationships, not as a separate relationship type. The validation layer must enforce: (1) no relationship between two relationships; (2) junction-connected paths only where direct relationship is permitted; (3) aggregation/composition to a relationship only from a composite element; (4) assignment direction always from active structure toward behavior or passive structure.

## 6. Motivation Elements

- **File Path**: `archimate-spec-3.2/ch-motivation-elements.html`
- **Semantic Summary**: Motivation elements model the "why" behind an Enterprise Architecture by representing the drivers, goals, principles, requirements, constraints, and stakeholder concerns that guide architectural design and change. They form a directed chain from external/internal conditions (drivers) through stakeholder interests, assessments, goals, outcomes, and principles down to concrete requirements and constraints, with meaning and value serving as interpretive overlays on any concept in the language.
- **Key Entities**: Stakeholder, Driver, Assessment, Goal, Outcome, Principle, Requirement, Constraint, Meaning, Value; relationships used within and between motivation elements: Association (Stakeholder → Driver, Stakeholder → Goal, Stakeholder ↔ Meaning, Stakeholder ↔ Value, Meaning ↔ any concept, Value ↔ any concept), Influence (Assessment → Assessment, Assessment → Goal, Goal → Goal, Outcome → Outcome, etc.), Realization (Requirement → Goal, Requirement → Outcome, Principle → Requirement, Constraint → Requirement, Goal → Outcome); relationships to core elements: Realization (core structure/behavior element → Requirement), Influence (core structure/behavior element → any motivation element), Assignment (business internal active structure element → Stakeholder)
- **Logic & Constraints**: Every motivation element type permits composition, aggregation, and specialization with elements of the same type (this is implicit and applies universally across the language). A stakeholder associates to drivers (their "concerns") and may influence other stakeholders; a driver need not have an associated stakeholder (external drivers exist independently). An assessment is always linked to a driver (it analyzes that driver) and may influence other assessments or goals, including negatively. A goal is associated with or realized by an outcome; goals may be decomposed into sub-goals via aggregation/composition. An outcome is the end result realized by capabilities; outcomes can be positive or negative (e.g., unwanted effects to mitigate); outcomes are related to goals but are distinct — a goal is what you want, an outcome is what you get. A principle is broader and more abstract than a requirement: a principle defines a general property for any system in a context; a requirement defines a property for a specific system described by the architecture. A constraint is distinguished from a requirement: a constraint restricts how a system operates or is realized (time, budget, technology, legal), not what functionality it must provide. Meaning can be associated with any concept (element or relationship) in the language; to express that a meaning is specific to a stakeholder, that stakeholder is also associated with the meaning. Value can be associated with any concept; to model the stakeholder for whom the value applies, that stakeholder is also associated with the value. Cross-layer rules for relating motivation to core elements: (1) a requirement (and, by transitivity via realization chains, also a principle, outcome, and goal) may be related to a structure or behavior element by means of a realization relationship (core element realizes the requirement); (2) the weaker influence relationship is permitted between motivation elements and structure or behavior elements; (3) meaning and value may be associated with any structure or behavior element; (4) a business internal active structure element (business actor, business role, or business collaboration) may be assigned to a stakeholder to express that an operational role within the enterprise is also a stakeholder. The full normative set of permitted relationships for all motivation element combinations is in Appendix B.
- **Requirements Context**: All ten motivation element types (Stakeholder, Driver, Assessment, Goal, Outcome, Principle, Requirement, Constraint, Meaning, Value) must be concrete instantiable classes under the abstract MotivationElement base defined in Chapter 4. The Goal/Outcome distinction (intent vs. result) must be preserved as separate types — they are not interchangeable even though both relate to desired states. The Principle/Requirement/Constraint hierarchy (decreasing abstraction, increasing specificity) implies a natural ordering for validation: a requirement may realize a principle, and a constraint imposes additional restrictions on a system independently of its functional requirements. Meaning and Value have a unique semantic: they associate with `Concept` (the full union type — element or relationship), not just `Element`; the library's association mechanism for these two types must accept any concept as the associated target. The assignment rule from business internal active structure to Stakeholder is a cross-layer relationship constraint the validation layer must enforce: only business actors, business roles, and business collaborations are valid sources for that assignment; no other core element type may be assigned to a Stakeholder. Influence between motivation elements must carry the optional sign/strength attribute (inherited from the general Influence relationship in Chapter 5).

## 7. Strategy Layer

- **File Path**: `archimate-spec-3.2/ch-strategy-layer.html`
- **Semantic Summary**: The Strategy Layer provides a small, deliberately minimal set of elements for modeling the strategic direction and choices of an enterprise — what it intends to do, what it is able to do, what assets it has to work with, and how it creates value — at an abstraction level above the operational Business Layer. Unlike the Business, Application, and Technology layers, the Strategy Layer collapses the active/passive structure distinction and omits service and interface elements entirely, reflecting that strategy concerns itself with "what" and "why" rather than "how" things are implemented or exposed.
- **Key Entities**: Resource, Capability, Value Stream, Course of Action
- **Logic & Constraints**: Resource is classified as a structure element that is neither active nor passive; it is realized by active or passive structure elements from the core layers. Capability and Value Stream are strategy behavior elements; Course of Action is a behavior element but is explicitly noted as not a "strategy behavior element" because it models a different type of behavior with a different relationship set. Resource, Capability, Value Stream, and Course of Action all permit composition, aggregation, and specialization with elements of the same type (universal rule). Capability may have serving relationships to other capabilities (one capability contributes to another). Capability is realized by core elements (internal and external behavior elements, people, processes, systems, and so on); a grouping can be used to denote that a set of core elements together realizes a capability. Capability increments are modeled using the specialization relationship (a specific version of a capability); capability increments and the core elements realizing them can be aggregated into plateaus to model evolution over time. Value Stream is composed of or aggregated from value stream stages (also modeled as Value Stream elements); stages are typically related using flow relationships. Resources can be assigned to value streams; capabilities can serve (enable) value streams. Value Stream is realized by business processes and possibly other core behavior elements; value stream stages provide a framework for organizing business processes. A Course of Action realizes or influences capabilities and resources; it represents what an enterprise has decided to do and may be categorized as a strategy (long-term, broad) or tactic (shorter-term, narrower). Cross-layer/cross-aspect relationship rules from Section 7.6: internal and external behavior elements (core) may realize Capabilities and Value Streams; active or passive structure elements (core) may realize a Resource; Capabilities, Value Streams, Courses of Action, and Resources may realize or influence Requirements (and indirectly, via derivation chains, also Principles or Goals); a Course of Action may also realize or influence an Outcome (and, indirectly, a Goal). The full normative permitted-relationship table for strategy elements is in Appendix B.
- **Requirements Context**: All four strategy element types (Resource, Capability, Value Stream, Course of Action) must be concrete instantiable classes. Resource must be classified as a structure element (not active, not passive) — this is an unusual placement that may require a dedicated abstract intermediate class `StrategyStructureElement` or a classification flag distinct from `ActiveStructureElement` and `PassiveStructureElement`. Capability and Value Stream must be classified as strategy behavior elements; Course of Action must be classified as a behavior element but excluded from the strategy behavior element subtype — the library's type hierarchy must represent this distinction. The serving relationship from Capability to Capability is a same-type relationship the validation layer must permit. The realization chain from core behavior elements to Capability/Value Stream and from core structure elements to Resource is a cross-layer constraint the validation layer must enforce directionally. The capability-increment pattern (specialization + plateau aggregation) is an important modeling idiom; the library should not block specialization on Capability. Value Stream's stage decomposition via composition/aggregation of Value Stream elements, connected by flow relationships, must be representable without special-casing. The distinction between Value Stream (strategic, value-perspective, no time-ordered task semantics) and business process (operational, sequence-oriented) is a semantic boundary that should be reflected in documentation and, where possible, in type-level constraints to prevent accidental conflation.

## 8. Business Layer

- **File Path**: `archimate-spec-3.2/ch-business-layer.html`
- **Semantic Summary**: The Business Layer models the operational organization of an enterprise in a technology-independent manner, defining the active entities (actors, roles, collaborations) that perform behavior, the behavioral elements (processes, functions, interactions, events, services) that constitute that behavior, and the passive information concepts (business objects, contracts, representations) on which behavior operates. It introduces service-orientation as the core design principle separating externally visible behavior (business service) from internal realization (process, function, interaction), and defines Product as the single cross-layer composite element that bundles services and passive structure from any layer with a contract for delivery to customers.
- **Key Entities**: Active structure — Business Actor, Business Role, Business Collaboration, Business Interface; Behavior — Business Process, Business Function, Business Interaction, Business Event, Business Service; Passive structure — Business Object, Contract, Representation; Composite — Product
- **Logic & Constraints**: "Business Internal Active Structure Element", "Business Internal Behavior Element", and "Business Passive Structure Element" are abstract; only their named specializations listed above are instantiable in models. Business Actor is assigned to one or more Business Roles; a Business Actor may be aggregated in a Location. Business Role is assigned to Business Processes or Business Functions; a Business Interface or Application Interface may serve a Business Role; a Business Interface may be part of (composed into) a Business Role. Business Collaboration aggregates two or more business internal active structure elements (roles, actors, or other collaborations) and may be assigned to Business Interactions or other internal behavior elements; a Business Interface may serve or be part of a Business Collaboration. Business Interface is composed into or aggregated by a Business Role or Actor and may be assigned to one or more Business Services to expose them. Business Process groups behavior by sequence/flow of activities toward a result; it may be triggered by or trigger any business behavior element (event, process, function, or interaction); it may access business objects; it may realize one or more business services; it may be served by business, application, or technology services; a business role may be assigned to it; an automated process may be realized by an application process. Business Function groups behavior by required resources, skills, or competencies (not by sequence); it has the same trigger, access, realization, and serving relationships as Business Process; a business role may be assigned to it; business functions can realize Capabilities. Business Interaction is performed collectively by a collaboration of two or more business actors, roles, or collaborations (a single active structure element is insufficient); it has the same trigger, access, realization, and serving relationships as Business Process; a business collaboration or two or more actors/roles may be assigned to it. Business Event is instantaneous (no duration); it may have a time attribute denoting when it occurs; it may trigger or be triggered by a business process, function, or interaction; it may access a business object; it may be composed of other business events. Business Service is the externally visible behavior element; it is realized by business processes, functions, or interactions; a business interface may be assigned to it; it may serve business, application, and technology behavior or active structure elements; it may access business objects. Business Object models a concept (type, not instance) in a business domain; it is passive and does not trigger or perform behavior; it may be accessed (created, read, written) by business processes, functions, interactions, events, or services; it may be realized by a Representation or by a Data Object (or both); it may have association, specialization, aggregation, or composition relationships with other business objects. Contract is a specialization of Business Object representing a formal or informal agreement between provider and consumer specifying rights, obligations, and functional/non-functional parameters for a product; all relationships valid for Business Object are also valid for Contract; additionally a Contract may have an aggregation relationship with a Product. Representation is a perceptible form (message, document) carrying the information of a business object; it realizes one or more business objects; a meaning may be associated with a representation. Product is a composite element that aggregates or composes business services, application services, technology services, business objects, data objects, technology passive structure elements (artifacts and material), and a contract — spanning across layers; a value may be associated with a product; a contract is expected to accompany a product but the relationship is aggregation (not mandatory composition). The many-to-many relationship between Business Process and Business Function is explicit: a process may be composed of functions and vice versa; processes describe flow, functions describe resource/skill groupings. Composition and aggregation between processes, functions, and interactions (same-type decomposition) are all permitted. Every element permits composition, aggregation, and specialization with elements of the same type (universal rule). The full normative set of permitted relationships is in Appendix B.
- **Requirements Context**: The thirteen concrete Business Layer element types must each be a directly instantiable class under the appropriate abstract intermediate base (BusinessInternalActiveStructureElement, BusinessInternalBehaviorElement, BusinessPassiveStructureElement). Contract must be modeled as a subclass of BusinessObject, not as an independent class; all BusinessObject validation rules must apply to Contract by inheritance, with the additional constraint that Contract may aggregate into Product. The Business Process / Business Function distinction (sequence-ordered vs. resource-grouped) must be preserved as separate types even though they share the same relationship surface — conflating them loses architectural intent. Business Interaction carries the same multiplicity constraint as the generic Interaction (Chapter 4): assignment requires either a BusinessCollaboration or two or more BusinessActor/BusinessRole elements; the validation layer must enforce this minimum-two constraint. Business Event's time attribute is a type-specific optional attribute; the per-element attribute schema infrastructure established in Chapter 4 must support it. Business Service is the only external behavior element in this layer and is the realization target for all three internal behavior types (Process, Function, Interaction) — the validation layer must permit all three as valid realizers. Representation realizes BusinessObject (not the reverse); a single Representation may realize multiple BusinessObjects; a single BusinessObject may have multiple Representations — the library's realization relationship must support this many-to-many pattern. Product is the sole cross-layer composite element in the Business Layer; its aggregation/composition targets span Business, Application, and Technology layers, so the library must not restrict Product's member types to Business Layer elements only. The Business Actor → Business Role → Business Process/Function assignment chain is the primary active-structure-to-behavior chain and should be validated directionally. BusinessFunction realizing Capability (from Chapter 7) is a cross-layer realization the validation layer must permit.

## 9. Application Layer

- **File Path**: `archimate-spec-3.2/ch-application-layer.html`
- **Semantic Summary**: The Application Layer provides a structurally parallel set of elements to the Business Layer for modeling application architecture: active structure elements (Application Component, Collaboration, Interface) that encapsulate and expose functionality, behavior elements (Function, Process, Interaction, Event, Service) that describe internal and external automated behavior, and a single passive structure element (Data Object) that represents typed, structured data for automated processing. Service-orientation is the governing design principle, with Application Service as the only external behavior element and Application Function/Process/Interaction as its internal realizers, accessed through Application Interfaces composed into Application Components.
- **Key Entities**: Active structure — Application Component, Application Collaboration, Application Interface; Behavior — Application Function, Application Interaction, Application Process, Application Event, Application Service; Passive structure — Data Object
- **Logic & Constraints**: "Application Internal Active Structure Element" and "Application Internal Behavior Element" are abstract; only the nine named specializations listed above are directly instantiable in models. Application Component is a self-contained, independently deployable unit; it performs one or more Application Functions and exposes services through Application Interfaces; it may be assigned to Application Functions or Application Processes; Application Components may realize other Application Components (abstraction levels per Section 3.6). Application Collaboration aggregates two or more Application Components or other Application Collaborations (minimum two); it is assigned to Application Interactions or other Application Internal Behavior Elements; an Application Interface may serve an Application Collaboration or be composed into it. Application Interface is composed into an Application Component (the component owns/provides the interface); it may be assigned to one or more Application Services (exposing them to the environment); it may also serve another Application Component or a node; the same service may be exposed through different interfaces and the same interface may expose multiple services. Application Function describes criteria-based internal behavior of an Application Component (automated behavior); it may realize Application Services; Application Services of other Application Functions and Technology Services may serve an Application Function; it may access Data Objects; an Application Component may be assigned to it; name convention: verb ending in "-ing". Application Interaction describes collective behavior performed by two or more Application Components via an Application Collaboration; an Application Collaboration (or two or more Application Components) may be assigned to it; it may realize Application Services; Business, Application, and Technology Services may serve it; it may access Data Objects. Application Process describes a sequence-ordered series of Application Behaviors (analogous to Business Process); an Application Component may be assigned to it; it may realize Application Services; other Application Services may serve it; it may access Data Objects; name convention: verb or verb-noun combination. Application Event is instantaneous (no duration); it may have a time attribute denoting when it occurs; it may trigger or be triggered (raised) by an Application Function, Process, or Interaction; it may access a Data Object; it may be composed of other Application Events; events may originate externally (from outside the organization) or internally. Application Service is the only external behavior element in this layer; it is realized by Application Functions or Application Processes; an Application Interface may be assigned to it (exposing it); it may serve Business, Application, and Technology behavior or active structure elements; it may access Data Objects; it should have business relevance and may have an associated Value element. Data Object models data structured for automated processing; it represents a logical type, not an instance (consistent with the ArchiMate type-not-instance convention); it may be accessed by Application Functions, Application Interactions, Application Processes, and Application Services; it may realize a Business Object (upward cross-layer realization) and may be realized by an Artifact (downward cross-layer realization); it may have association, specialization, aggregation, or composition relationships with other Data Objects; a Data Object should be self-contained with clear business meaning, not merely technical. Every element in the layer permits composition, aggregation, and specialization with elements of the same type (universal rule). The full normative set of permitted relationships is in Appendix B.
- **Requirements Context**: The nine concrete Application Layer element types must each be directly instantiable classes under the appropriate abstract intermediate base (ApplicationInternalActiveStructureElement, ApplicationInternalBehaviorElement, ApplicationPassiveStructureElement). Application Component is the structural anchor of the layer and should be the canonical active structure class; its realization of other Application Components must be permitted by the validation layer. Application Collaboration carries the same minimum-two multiplicity constraint as the generic Collaboration (Chapter 4): assignment requires an ApplicationCollaboration or two or more ApplicationComponent elements — the validation layer must enforce this. Application Interface has a dual role: it is both composed into a Component (owned interface) and assigned to Services (exposure relationship) — these are two distinct relationships and the library must not conflate them. Application Event carries the optional time attribute, consistent with Business Event; the per-element attribute schema infrastructure must support it. Application Service is the sole external behavior element and the only valid realization target from Application Function, Application Process, and Application Interaction — the validation layer must permit all three as valid realizers. Data Object's cross-layer realization pattern is bidirectional in direction of abstraction: Data Object realizes Business Object (Application → Business, upward) and is realized by Artifact (Technology → Application, downward); the validation layer must permit both directions and must not restrict realization targets to same-layer elements. Data Object models types not instances — documentation and any schema validation should reinforce this; the library should not provide instance-management semantics for Data Object.

## 10. Technology Layer

- **File Path**: `archimate-spec-3.2/ch-technology-layer.html`
- **Semantic Summary**: The Technology Layer models the technology infrastructure of the enterprise using a set of elements structurally parallel to the Business and Application Layers — active structure elements (Node, Device, System Software, Technology Collaboration, Technology Interface, Path, Communication Network), behavior elements (Technology Function, Technology Process, Technology Interaction, Technology Event, Technology Service), and passive structure (Artifact) — with service-orientation as the governing design principle separating internal technology behavior from externally exposed technology services. The layer is extended by a Physical sub-set (Equipment, Facility, Distribution Network, Material) that models the tangible, non-IT physical world and may be combined with IT infrastructure elements within the same node to represent integrated operational-technology systems.
- **Key Entities**: IT active structure — Node, Device, System Software, Technology Collaboration, Technology Interface, Path, Communication Network; IT behavior — Technology Function, Technology Process, Technology Interaction, Technology Event, Technology Service; IT passive structure — Artifact; Physical active structure — Equipment, Facility, Distribution Network; Physical passive structure — Material
- **Logic & Constraints**: Node is the primary active structure element of the Technology Layer; it represents a computational or physical resource that hosts, manipulates, or interacts with other computational or physical resources; nodes may consist of sub-nodes; a node may be composed of or aggregate devices, system software, facilities, and equipment to model an integrated piece of IT and operational technology. Device is a specialization of Node representing a physical IT resource (hardware: mainframe, PC, router, virtualized hardware in IaaS); artifacts and system software may be deployed on a device; devices are interconnected by communication networks; a device may be assigned to artifacts to model deployment; devices may be composite. System Software represents the software environment (OS, JEE server, database, workflow engine, middleware) in which artifacts run; a device or system software can be assigned to other system software to model layered software stacks; system software can be assigned to artifacts to model deployment; system software may realize other system software; a node may be composed of or aggregate system software. Technology Collaboration aggregates two or more technology internal active structure elements (minimum two); it may be assigned to technology interactions or other technology internal behavior elements; a technology interface may serve or be composed into a technology collaboration. Technology Interface is the point of access where technology services offered by a technology internal active structure element can be accessed; it is composed into a technology internal active structure element (owner/provider); it may be assigned to one or more technology services (exposing them); the same service may be exposed through different interfaces; formerly called "infrastructure interface" (deprecated). Path represents the logical communication (or distribution) link between two or more technology internal active structure elements through which they can exchange data, energy, or material; it is realized by one or more communication networks (or distribution networks for physical elements); properties such as bandwidth and latency are typically aggregated from the underlying networks; a path can aggregate nodes. Communication Network represents the physical communication infrastructure (hardware and software elements for networking and physical communications); it connects two or more devices or system software; it realizes one or more paths; it may consist of sub-networks; it can aggregate devices and system software to model routers, switches, and firewalls; formerly called "network" (deprecated). Technology Function is criteria-based internal behavior of a technology internal active structure element (invisible to users of that element); it may realize technology services; it may access technology passive structure elements; a technology internal active structure element may be assigned to it; formerly called "infrastructure function" (deprecated); name convention: verb ending in "-ing". Technology Process is sequence-ordered internal behavior achieving a specific result; it may realize technology services; other technology services may serve it; a technology internal active structure element may be assigned to it; it may access technology passive structure elements; name convention: verb or verb-noun (e.g., "Boot up system", "Replicate database"). Technology Interaction is collective behavior performed by two or more technology internal active structure elements (minimum two); a technology collaboration or two or more nodes may be assigned to it; it may realize technology services and technology services may serve it; it may access artifacts. Technology Event is instantaneous (no duration); it may have a time attribute denoting when it occurs (e.g., for scheduled recurring infrastructure functions); it may trigger or be triggered by technology functions, processes, or interactions; it may access an artifact or material; it may be composed of other technology events; events may originate externally or internally. Technology Service is the only external behavior element in the IT Technology Layer; it exposes the functionality of a technology internal active structure element to its environment (application components, nodes); it is realized by technology functions, processes, or interactions; it is exposed via technology interfaces assigned to it; it may serve business, application, and technology behavior or active structure elements; it may access artifacts; typical examples include messaging, storage, naming, and directory services; formerly called "infrastructure service" (deprecated). Artifact is a "physical" element in the IT world (a specialization of technology passive structure element); it models software products such as source files, executables, scripts, database tables, messages, documents, specifications, and model files; an application component or system software may be realized by one or more artifacts; a data object may be realized by one or more artifacts; a device or system software element may be assigned to an artifact to model deployment; the two canonical uses are as an execution component or as a data file; artifacts may consist of sub-artifacts. Physical elements extend the Technology Layer with active and passive structure only — no specific physical behavior elements are defined; physical technology elements may be combined with IT elements (e.g., device) and form part of the same node to model integrated operational-technology. Equipment represents one or more physical machines, tools, or instruments that can create, use, store, move, or transform materials; material may be accessed (created, used, moved, transformed) by equipment; equipment may be assigned to material to model where material is stored; material may realize equipment; equipment may serve other equipment and other active structure elements (e.g., business roles and actors); facilities may be assigned to equipment (equipment is installed at/on a facility); equipment may be composed of other equipment or devices; equipment may be aggregated in a location. Facility represents a physical structure or environment (factory, building, warehouse, outdoor construction) that facilitates the use of equipment; it may be composed of other facilities or nodes; it may be interconnected by distribution networks; it can serve other facilities and active structure elements (business roles and actors); it may be aggregated in a location. Distribution Network represents the physical distribution or transportation infrastructure; it is the physical realization of the logical path between facilities; it connects two or more facilities and may realize one or more paths; it may consist of sub-networks; it can aggregate facilities and equipment (e.g., railway stations and trains in a rail network). Material represents tangible physical matter or energy (raw materials, physical products, energy sources such as fuel and electricity); it may be accessed by technology behavior elements; equipment may be assigned to material to model where it is stored; material may realize equipment; material may be composed of other material.
- **Requirements Context**: The twelve concrete Technology Layer element types (Node, Device, System Software, Technology Collaboration, Technology Interface, Path, Communication Network, Technology Function, Technology Process, Technology Interaction, Technology Event, Technology Service, Artifact) plus the four physical element types (Equipment, Facility, Distribution Network, Material) must each be directly instantiable classes under the appropriate abstract intermediate bases (TechnologyInternalActiveStructureElement, TechnologyInternalBehaviorElement, TechnologyPassiveStructureElement, and their physical counterparts). Node is the structural anchor of the IT side; Device and System Software are specializations of Node and are commonly composed together inside a Node to form a general-purpose host — the library must permit Node to contain both as compositions. The Technology Collaboration minimum-two multiplicity constraint (same as the generic Collaboration rule from Chapter 4) must be enforced by the validation layer. Technology Interface has the same dual role as Application Interface: it is composed into a technology internal active structure element (ownership) and independently assigned to Technology Services (exposure) — these are two distinct relationships that must not be conflated. Technology Event carries the optional time attribute (consistent with Business Event and Application Event) — the per-element attribute schema must support it; Technology Event is the only behavior element permitted to access Material (as well as Artifact), which constrains the access-target validation rule. Technology Service is the sole external behavior element and the only valid realization target from Technology Function, Technology Process, and Technology Interaction — the validation layer must permit all three as valid realizers. Artifact's cross-layer realization pattern is bidirectional in the abstraction hierarchy: an Application Component or System Software may be realized by Artifact (downward, Technology realizes Application), and a Data Object may be realized by Artifact (downward, Technology realizes Application) — the validation layer must permit both. Path and CommunicationNetwork form a logical-physical pair: Path is the logical link between technology internal active structure elements, CommunicationNetwork is its physical realization; this realization relationship (CommunicationNetwork realizes Path) must be permitted by the validation layer. Similarly, DistributionNetwork realizes Path for the physical domain. The physical elements (Equipment, Facility, DistributionNetwork, Material) have no dedicated physical behavior element type; they rely on Technology behavior elements (Technology Function, Technology Process, etc.) for behavior modeling — the library must not restrict technology behavior assignment to IT elements only but must also permit assignment from physical active structure elements. Equipment's assignment-to-Material relationship (models where material is stored) is a non-standard use of assignment (active structure assigned to passive structure) that the validation layer must explicitly permit. Material's ability to realize Equipment is an unusual passive-to-active realization that the validation layer must accommodate. The deprecated former names ("infrastructure interface", "infrastructure function", "infrastructure service", "network") should be noted in library documentation but need not be modeled as distinct types; the current names are normative.

## 11. Relationships Between Core Layers

- **File Path**: `archimate-spec-3.2/ch-relationships-between-core-layers.html`
- **Semantic Summary**: This chapter defines the normative cross-layer relationship patterns that express business-IT alignment, specifying which serving and realization relationships are permitted between the Business, Application, and Technology layers and establishing that indirect relationships between non-adjacent layers can be derived through chains of direct relationships. It treats alignment as bidirectional — both upward (application and technology support business) and downward (business behavior can be realized by application and technology elements) — while prohibiting the realization of business active structure elements (people) by application or technology elements.
- **Key Entities**: Cross-layer Serving relationships (Application Service → Business behavior, Application Interface → Business Role, Business Service → Application behavior, Business Interface → Application Component, Technology Service → Application behavior, Technology Interface → Application Component, Application Service → Technology behavior, Application Interface → Technology internal active structure); Cross-layer Realization relationships (Application process/function → Business process/function, Data Object → Business Object, Technology passive structure → Business Object, Technology process/function → Application process/function, Artifact → Data Object, Artifact → Application Component); Product aggregation (Product ↔ Application/Technology Service, Product ↔ Data Object/Technology passive structure); derived indirect Business-Technology relationships
- **Logic & Constraints**: Between the Business Layer and lower layers, two categories of cross-layer relationship are permitted. First, serving relationships in both directions: application services and application interfaces may serve business behavior elements and business roles respectively; conversely, business services and business interfaces may serve application behavior elements and application components. Second, realization relationships from lower to higher: an application process or function may realize a business process or function; a data object or technology passive structure element may realize a business object to indicate that the data object is a digital representation of the corresponding business object, or the technology element is a physical representation. There is an explicit prohibition on realizing business internal active structure elements (Business Actor, Business Role, Business Collaboration) by application or technology elements, because people cannot be realized by applications or technology; instead, the business behavior of those active structure elements may be realized by application or technology behavior elements, to which application or technology active structure elements can in turn be assigned. Between the Application Layer and the Technology Layer, the same two categories apply: serving relationships in both directions (technology service/interface → application behavior/component, and application service/interface → technology behavior/internal active structure), and realization relationships upward (technology process/function → application process/function, artifact → data object to indicate a physical data file is a digital realization, artifact → application component to indicate a physical executable realizes an application or part of one). An artifact realizes an application component and is itself deployed on a device or system software element via assignment — this is the canonical deployment modeling pattern. Product may have aggregation relationships with application services, technology services, data objects, and technology passive structure elements (artifacts, material) to indicate those services or objects are offered directly to a customer as part of the product. Derived indirect relationships between non-adjacent layers are valid: if a business object is realized by a data object, which in turn is realized by an artifact, then the artifact indirectly realizes the business object; these transitive chains follow the derivation rules of Section 5.7 and may be modeled directly without the intermediary elements being present.
- **Requirements Context**: The validation layer must enforce the prohibition on realizing business internal active structure elements (BusinessActor, BusinessRole, BusinessCollaboration): no realization relationship may have any of these three types as its target. The cross-layer serving relationship surface is bidirectional across all three layer pairs and must be explicitly permitted by the validation layer for each of: ApplicationService → BusinessBehaviorElement, ApplicationInterface → BusinessRole, BusinessService → ApplicationBehaviorElement, BusinessInterface → ApplicationComponent, TechnologyService → ApplicationBehaviorElement, TechnologyInterface → ApplicationComponent, ApplicationService → TechnologyBehaviorElement, ApplicationInterface → TechnologyInternalActiveStructureElement. The cross-layer realization relationship surface must permit: ApplicationProcess/ApplicationFunction → BusinessProcess/BusinessFunction; DataObject → BusinessObject; TechnologyPassiveStructureElement → BusinessObject; TechnologyProcess/TechnologyFunction → ApplicationProcess/ApplicationFunction; Artifact → DataObject; Artifact → ApplicationComponent. The Product aggregation rule (Product may aggregate ApplicationService, TechnologyService, DataObject, and TechnologyPassiveStructureElement) must be supported without restriction to Business Layer members — this is a cross-layer composite pattern consistent with the Chapter 8 requirement. The derivation engine must support transitive chain resolution through the realization relationship across all three layers so that indirect Business-Technology relationships (e.g., BusinessObject ← DataObject ← Artifact) can be surfaced; this requires the engine to traverse multi-hop realization chains, not only adjacent-layer pairs.

## 12. Implementation and Migration Layer

- **File Path**: `archimate-spec-3.2/ch-implementation-and-migration-layer.html`
- **Semantic Summary**: The Implementation and Migration Layer provides a dedicated set of elements for modeling the planning and execution of architectural change — projects, deliverables, events, and transition states — that sits outside the core three-layer framework but can be related to all core elements. It treats architecture change as a first-class concern by distinguishing the time-bounded work that drives change (Work Package), the concrete outputs produced (Deliverable), the instantaneous state changes that mark progress (Implementation Event), the stable architectural states that anchor migration plans (Plateau), and the differences between those states (Gap).
- **Key Entities**: Work Package (behavior element), Deliverable (passive structure element), Implementation Event (behavior element), Plateau (composite element), Gap (other element — modeled as an association between two plateaus)
- **Logic & Constraints**:
  - **Work Package vs. Business Process**: A work package is conceptually similar to a business process — both consist of causally related tasks aimed at a well-defined result — but a work package is a unique "one-off" process, not a continuously repeating activity. It has a start and end (modeled as attributes) and may represent sub-projects, complete projects, programs, portfolios, or Agile iterations (sprints) or increments.
  - **Deliverable as passive structure**: A deliverable is classified as a passive structure element — it is the precisely defined result of a work package and may be any type of output (reports, software, physical products, organizational change, architecture implementations). Deliverables are typically contractually specified and formally reviewed and signed off by stakeholders.
  - **Implementation Event instantaneous nature and time attribute**: Unlike a work package, an implementation event has no duration — it is instantaneous, representing a state change related to implementation or migration. It may carry a time attribute denoting the moment or moments at which it occurs, enabling modeling of project schedules, milestones, project completions, and lifecycle changes of deliverables. The name should preferably be a verb in the perfect tense (e.g., "project initiation phase completed"). An implementation event may trigger or be triggered by a work package or a plateau, may access a deliverable, and may be composed of other implementation events. It may be associated with any core element to indicate a lifecycle state change.
  - **Plateau as composite element spanning core concepts**: A plateau represents a relatively stable state of the architecture during a limited period of time, corresponding to the Baseline, Target, and Transition Architecture states in the TOGAF ADM. A plateau may aggregate or compose any concept of the ArchiMate core language to indicate which parts of the architecture belong to it. Realization from a plateau to parts of the architecture is also permitted (e.g., a capability may be realized by a plateau to signify that a capability increment is valid only during that plateau's time span). Motivation elements (goals, outcomes, requirements) may be aggregated or composed in plateaus to indicate their applicability to specific architectural states. A plateau may trigger or be triggered by an implementation event.
  - **Gap as a two-plateau association**: A gap is associated with exactly two plateaus (e.g., Baseline and Target, or two consecutive Transition Architectures) and represents the differences between them. A gap may also be associated with the individual core concepts that are unique to one of the two linked plateaus — the concepts that constitute the actual differences. In the TOGAF framework, a gap is a key output of gap analysis in ADM Phases B, C, and D and forms a critical input to subsequent implementation and migration planning.
  - **Deprecated WorkPackage→Deliverable realization relationship**: The realization relationship from Work Package to Deliverable (shown with an asterisk in Figure 106, the Implementation and Migration Metamodel) is deprecated and may be removed in a future version of the standard. It is recommended to use an access relationship instead.
  - **Section 12.4 — Relationships**: Implementation and migration elements use the standard ArchiMate relationships. No additional relationship types are defined for this layer.
  - **Section 12.5 — Relationships with Other Aspects and Layers**: A business internal active structure element (e.g., a Project Manager business role) may be assigned to a work package. A plateau may aggregate, compose, or realize any concept of the ArchiMate core language. A deliverable may realize any concept of the ArchiMate core language (including requirements and core architecture elements) to indicate that the deliverable implements that concept. A gap is associated with the core concepts that make up the difference between its two linked plateaus. A composite element may aggregate a work package or deliverable. The association relationship may be used to show that parts of the architecture are affected in some way by certain work packages. Relationships between implementation and migration elements and motivation elements (e.g., a deliverable realizing a requirement or goal) are technically indirect (through the realization of a core element), but may be made explicit in a model for clarity. Motivation elements (goals, outcomes, requirements) may be aggregated or composed in plateaus to model their applicability to specific architectural states; plateaus may also be used for capability-based planning via aggregation/composition.
- **Requirements Context**: The five implementation and migration element types must be directly instantiable classes with clear abstract base assignments: WorkPackage and ImplementationEvent as behavior elements, Deliverable as a passive structure element, Plateau as a composite element (parallel to Grouping and Location from Chapter 4), and Gap as a distinct associative element that carries two mandatory plateau references. The deprecation of the WorkPackage→Deliverable realization relationship must be reflected in the library — the validation layer should emit a deprecation warning when this relationship is used and documentation must recommend the access relationship as the replacement. Plateau's ability to aggregate or compose any core concept makes it structurally similar to Grouping but semantically distinct (it denotes a time-bounded architectural state, not an organizational container) — the library must represent these as separate types and must not restrict Plateau's member types to implementation and migration elements only. Gap's mandatory two-plateau association is a structural constraint the validation layer must enforce: a Gap is invalid unless associated with exactly two Plateau elements. ImplementationEvent's time attribute must be supported by the per-element attribute schema infrastructure (consistent with BusinessEvent, ApplicationEvent, and TechnologyEvent). The assignment rule permitting business internal active structure elements to be assigned to a work package is a cross-layer relationship constraint the validation layer must permit. Deliverable's cross-layer realization surface — it may realize any ArchiMate core concept including requirements, capabilities, and architecture elements — means the library must not restrict the realization target of Deliverable to implementation and migration elements only. The Work Package / Business Process conceptual similarity (same relationship surface, different lifecycle semantics) means care must be taken not to merge the two types; the "one-off vs. repeating" distinction is a semantic boundary that should be documented even if it cannot be automatically enforced.

## 13. Stakeholders, Architecture Views, and Viewpoints

- **File Path**: `archimate-spec-3.2/ch-stakeholders-architecture-views-and-viewpoints.html`
- **Semantic Summary**: This chapter defines the viewpoint mechanism — the formal procedure by which architects create architecture views tailored to specific stakeholder concerns — and establishes that the mechanism conforms to the ISO/IEC 42010 standard for Architecture Description. A viewpoint prescribes the concepts, models, analysis techniques, and visualizations that govern a view; a view is a partial, stakeholder-targeted depiction of the architecture produced by applying a viewpoint to an underlying ArchiMate model.
- **Key Entities**: Architecture View, Architecture Viewpoint, Stakeholder, Concern, Purpose dimension (Designing / Deciding / Informing), Content dimension (Details / Coherence / Overview), viewpoint mechanism, ISO/IEC 42010
- **Logic & Constraints**:
  - **View vs. Viewpoint distinction**: A view is what you see (a partial depiction of the architecture addressing specific concerns); a viewpoint is where you are looking from (the set of conventions that govern how a view is constructed, interpreted, and analyzed). A view is governed by exactly one viewpoint; a viewpoint may govern many views.
  - **Conformance to ISO/IEC 42010**: The ArchiMate viewpoint mechanism is defined to conform to ISO/IEC 42010, which provides the conceptual model of an Architecture Description including stakeholders, concerns, viewpoints, and views. This is a normative alignment requirement, not merely informative.
  - **Stakeholders and concerns**: An architecture view addresses one or more concerns held by a stakeholder of the system. A concern can be framed by more than one viewpoint. A viewpoint frames one or more concerns. Relevance to a stakeholder's concern is the selection criterion for which elements and relationships appear in a view.
  - **Two-step viewpoint creation process**: (1) Select a subset of relevant concepts (elements and relationships) from the ArchiMate metamodel based on the information needed to address the stakeholder's concerns. (2) Define a representation to depict those concepts in a way understood by the stakeholders. A representation may be a diagram using standard or customized ArchiMate notation, a catalog of elements, a matrix showing relationships between groups of elements, or an entirely different visualization.
  - **Applying a viewpoint**: Applying a viewpoint to an architecture model means selecting the parts of the architecture that match the Step-1 concept set and depicting them in the manner prescribed by Step 2.
  - **3×3 purpose × content classification framework**: Viewpoints are classified along two dimensions — purpose and content — yielding nine possible classification cells.
    - Purpose categories: *Designing* (supports architects and designers in the design process from initial sketch to detailed design; typically produces diagrams), *Deciding* (assists managers by offering insight into cross-domain architecture relationships; typical forms are cross-reference tables, landscape maps, lists, and reports), *Informing* (helps any stakeholder achieve understanding, obtain commitment, or convince adversaries; typical forms are illustrations, animations, cartoons, or flyers).
    - Content categories: *Details* (typically considers one layer and one aspect from the ArchiMate Core Framework; typical stakeholders are software engineers or process owners), *Coherence* (spans multiple layers or multiple aspects to focus on cross-layer or cross-aspect relationships; typical stakeholders are operational managers), *Overview* (addresses both multiple layers and multiple aspects simultaneously; typically addressed to Enterprise Architects, CEOs, and CIOs).
  - **What a view must and may contain**: A view contains elements and relationships (concepts) from the ArchiMate metamodel selected by its governing viewpoint. The architect designs and creates an appropriate representation for those elements and relationships, suitable for the stakeholders and concerns being framed. The view does not have to be visual or graphical in nature.
  - **Representation format**: No single representation format is mandated. The architect may use the profile mechanism (Section 14.1) to create representations based on attributes of elements and relationships — for example, to produce color-coded heat maps. Diagrams, catalogs, matrices, and non-graphical formats are all explicitly permitted.
  - **Views have a limiting character**: A view is always a partial and incomplete depiction of the system; it restricts the whole architecture to a selected subset of aspects. Views are inter-related; combinations of views together with their underlying inter-dependency relationships are often the best way to describe and communicate architecture.
  - **Example viewpoints**: A catalogue of example viewpoints is provided in Appendix C. Support for example viewpoints is optional per Section 1.3.
- **Requirements Context**: Section 1.3 of the specification lists supporting the viewpoint mechanism (Chapter 13) as a mandatory conformance requirement for any conforming implementation — this is not optional. The Python library must therefore provide a first-class `Viewpoint` type that captures: (1) the subset of permitted ArchiMate concept types (elements and/or relationships from the metamodel) that may appear in views governed by it; (2) a purpose classification (Designing / Deciding / Informing); (3) a content classification (Details / Coherence / Overview); and (4) an optional representation description. A `View` type must reference its governing `Viewpoint` and contain only concepts permitted by that viewpoint's Step-1 concept set. The library must not restrict the representation format of a view to diagrams — catalogs, matrices, and non-graphical representations are spec-valid. The ISO/IEC 42010 conformance alignment implies that `Stakeholder`, `Concern`, `Viewpoint`, and `View` form a coherent sub-model within the library, and that the associations between them (stakeholder → concerns, concern ↔ viewpoints, viewpoint → views) should be explicitly navigable. The 3×3 classification grid should be modeled as two enums (PurposeCategory, ContentCategory) on the Viewpoint type, not as free-form strings, to enable tooling-level classification and filtering. Because a view is always a partial depiction — never the whole model — the library should enforce that `View` is a projection or filter over an underlying `Model`, not an independent container.
